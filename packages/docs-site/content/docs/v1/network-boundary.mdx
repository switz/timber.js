# The Network Boundary

Every website is built across a network boundary. There's a server and there's a client. A request goes one way, a response comes back. Every framework in history has dealt with this — Rails, PHP, SPAs, universal SSR. Most pretend one side doesn't exist.

SPAs pretend the server doesn't exist — everything happens in the browser, and the server is just an API. Traditional server frameworks pretend the client doesn't exist — every interaction is a full page load. Universal SSR frameworks (Next.js, Remix) render on both sides but blur the line between them.

React Server Components make the boundary explicit. Server components run on the server. Client components run on both (server-rendered, then hydrated). The `'use client'` directive marks where you cross the boundary.

## Server Components

By default, every component in timber.js is a server component. It runs on the server, has access to the database, file system, secrets, and anything else on the server. It sends HTML to the client — no JavaScript.

```tsx
// This runs on the server. No JS shipped to the browser.
export default async function UserProfile({ params }) {
  const { id } = await params;
  const user = await db.users.findUnique({ where: { id } });
  return <h1>{user.name}</h1>;
}
```

## Client Components

Add `'use client'` at the top of a file to make it a client component. It renders on the server (for the initial HTML) and hydrates in the browser (for interactivity).

```tsx
'use client';
import { useState } from 'react';

export function Counter() {
  const [count, setCount] = useState(0);
  return <button onClick={() => setCount(c => c + 1)}>Count: {count}</button>;
}
```

Only use `'use client'` when you need browser APIs or interactivity. Every client component adds JavaScript to the bundle.

## The Boundary in Practice

Server components can import and render client components. Client components cannot import server components — they can only receive them as `children` or other props.

```tsx
// Server component — can use client components
import { Counter } from './counter'; // 'use client' file

export default async function Page() {
  const data = await fetchData(); // Server-only
  return (
    <div>
      <h1>{data.title}</h1>
      <Counter /> {/* Client component rendered inside server component */}
    </div>
  );
}
```

## How Streaming Fits In

The server renders the component tree and streams the result to the browser. Content outside `<Suspense>` renders before the flush — it determines the HTTP status code and headers. Content inside `<Suspense>` streams in after.

The flush point is where the server commits its response. Everything before it is "the shell" — the primary content that must be correct before the browser sees anything. Everything after it is secondary content that arrives progressively.

This is the key insight: **the flush point is the network boundary for HTTP semantics.** Above it, the server has full control. Below it, the response is already committed.

## Logging and Visibility

timber.js logs a request tree in dev mode showing exactly what happened on each side of the boundary:

- Which middleware ran and how long each took
- Which access checks passed or failed
- When the shell was ready and the flush happened
- Which Suspense boundaries are still pending
- Which client components were hydrated

This visibility is intentional. The network boundary shouldn't be mysterious. You should be able to see exactly what runs where and when.
