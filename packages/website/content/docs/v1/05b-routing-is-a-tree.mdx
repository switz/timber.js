# Your routing is a tree

timber's file-system routing isn't just a convenience for organizing files — it *is* the component tree. Every directory is a node. Every `layout.tsx` wraps its children. Every `page.tsx` is a leaf.

## The mapping

```
app/
  layout.tsx              → <RootLayout>
  page.tsx                →   <HomePage />
  blog/
    layout.tsx            →   <BlogLayout>
    page.tsx              →     <BlogIndex />
    [slug]/
      page.tsx            →     <BlogPost />
  settings/
    layout.tsx            →   <SettingsLayout>
    page.tsx              →     <SettingsPage />
```

When a user visits `/blog/hello-world`, timber assembles this tree:

```
<RootLayout>
  <BlogLayout>
    <BlogPost slug="hello-world" />
  </BlogLayout>
</RootLayout>
```

Layouts wrap. Pages fill in the leaf. The URL determines which branch of the tree to render.

## Segments are tree nodes

Each directory in your route is a **segment**. Segments are the building blocks:

```
/blog/hello-world
  ↓       ↓
 seg1    seg2
```

Each segment can have its own `layout.tsx`, `middleware.tsx`, `access.ts`, and `page.tsx`. These all compose at that node in the tree.

## Route groups are invisible nodes

Parenthesized directories like `(auth)` or `(marketing)` create nodes in the tree that don't appear in the URL:

```
app/
  (public)/
    layout.tsx          ← no-auth layout
    about/
      page.tsx          → /about
  (authenticated)/
    layout.tsx          ← requires login
    dashboard/
      page.tsx          → /dashboard
```

Route groups let you share layouts and access controls without affecting URLs. They're invisible branches in the tree.

## Parallel routes are siblings

Slots — directories prefixed with `@` — create sibling branches that render simultaneously:

```
app/
  dashboard/
    layout.tsx
    page.tsx
    @sidebar/
      page.tsx
    @notifications/
      page.tsx
```

The layout receives these as named props:

```tsx
export default function DashboardLayout({
  children,
  sidebar,
  notifications,
}: {
  children: React.ReactNode;
  sidebar: React.ReactNode;
  notifications: React.ReactNode;
}) {
  return (
    <div className="grid grid-cols-[250px_1fr_300px]">
      <aside>{sidebar}</aside>
      <main>{children}</main>
      <aside>{notifications}</aside>
    </div>
  );
}
```

Slots are parallel subtrees. They match the URL independently, have their own error boundaries, and can have their own access controls. They're siblings in the tree, rendered side by side.

## Dynamic segments are wildcards

Square brackets create parameterized nodes:

```
app/
  products/
    [id]/
      page.tsx          → /products/123, /products/abc
    [id]/
      reviews/
        page.tsx        → /products/123/reviews
```

The tree structure is the same — `[id]` is just a node that matches any value and captures it as a parameter.

## Matching priority

When multiple routes could match a URL, timber follows a clear priority:

1. **Static segments** — exact match wins (`/blog` beats `/[slug]`)
2. **Dynamic segments** — `[param]` matches one segment
3. **Catch-all** — `[...param]` matches one or more segments
4. **Optional catch-all** — `[[...param]]` matches zero or more

This is the same intuition as "more specific wins." A concrete path beats a wildcard.

## The tree is the architecture

Once you internalize that routing is a tree, the rest follows:

- **Layouts** wrap their children — they're inner nodes
- **Pages** are leaves — they render the content
- **Access controls** gate subtrees — deny at a node, block everything below
- **Error boundaries** catch at a node — errors bubble up the tree
- **Navigation** swaps subtrees — keep the shared trunk, replace the changed branch

The filesystem is not a metaphor for the architecture. It *is* the architecture.
