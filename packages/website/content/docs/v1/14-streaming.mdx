# Streaming

timber.js streams HTML, but only after the important stuff is settled. The status code, headers, and primary content are committed before the first byte leaves the server. Secondary content streams in via `<Suspense>`.

## The Flush Point

The response flushes when all three conditions are met:

1. All `middleware.ts` in the segment chain have completed
2. All `access.ts` checks have passed
3. React's `onShellReady` has fired — everything outside `<Suspense>` has rendered

Until then, no bytes are sent. Once they are, the correct status code is committed and the shell flushes. Suspense boundaries stream their content into the open connection.

## Placing Suspense Boundaries

You choose where the flush boundary sits. Content outside `<Suspense>` can affect the status code. Content inside cannot.

```tsx
import { Suspense } from 'react';
import { deny } from '@timber-js/app/server';

export default async function ProductPage({ params }) {
  const { id } = await params;
  const product = await getProduct(id);
  if (!product) deny(404); // Real HTTP 404

  return (
    <div>
      <ProductHeader product={product} />
      <Suspense fallback={<ReviewsSkeleton />}>
        <ProductReviews productId={product.id} />
      </Suspense>
    </div>
  );
}
```

`getProduct` resolves before the flush. If it fails, the server sends a real 404. Reviews are secondary content — they stream after the 200 is committed.

## No `loading.tsx`

There is no `loading.tsx` convention. timber.js does not insert Suspense boundaries for you. If you want a loading state, you place `<Suspense>` explicitly. This keeps you in control of what streams and what blocks.

## The Layout Suspense Footgun

Wrapping `children` in `<Suspense>` inside a layout surrenders HTTP correctness for every page in that subtree:

```tsx
// ⚠️ Don't do this
export default function Layout({ children }) {
  return (
    <Shell>
      <Suspense fallback={<Skeleton />}>{children}</Suspense>
    </Shell>
  );
}
```

The framework warns in dev mode when this is detected. For navigation loading states, use `useNavigationPending()` instead.

## `deferSuspenseFor`

Sometimes Suspense content resolves fast enough that you'd rather wait for it than show a fallback. `deferSuspenseFor` holds the SSR stream for a specified duration, giving fast boundaries time to resolve inline:

```tsx
export const deferSuspenseFor = 200; // milliseconds

export default async function ProductPage({ params }) {
  const { id } = await params;
  const product = await getProduct(id);
  if (!product) deny(404);

  return (
    <div>
      <ProductHeader product={product} />
      <Suspense fallback={<ReviewsSkeleton />}>
        <ProductReviews productId={product.id} />
      </Suspense>
    </div>
  );
}
```

If reviews resolve within 200ms, they render inline — no fallback ever shown. If they take longer, the fallback flushes and reviews stream in when ready.

## Early Hints (103)

Before the flush point, the framework sends 103 Early Hints with CSS, fonts, and JS that the browser will need. These fire at route-match time — before middleware, before rendering — based on the build manifest. The browser starts fetching critical resources while the server is still working.

Platform support varies: Cloudflare handles it automatically via `Link` headers. Node.js uses `res.writeEarlyHints()`. Serverless platforms may or may not pass 103 responses through.
