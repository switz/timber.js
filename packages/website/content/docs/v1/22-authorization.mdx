# Authorization

Authorization in timber.js lives in `access.ts` files co-located with route segments. Each segment that needs auth has its own access gate.

## `access.ts`

```ts
// app/(authenticated)/access.ts
import type { AccessContext } from '@timber-js/app/server';
import { cookies, redirect } from '@timber-js/app/server';

export default async function access(ctx: AccessContext) {
  const session = getSessionFromCookie(cookies());
  if (!session) redirect('/login');

  // Warm the cache for the layout below
  await getUser(session.userId);
}
```

`access.ts` is a gate — it passes or fails. Return values are discarded. If it doesn't throw or redirect, rendering proceeds.

## Sharing Data Between Access and Layout

Access checks and layouts share the same `React.cache` scope. Call the same cached function in both — the layout gets a free cache hit:

```tsx
// app/(authenticated)/layout.tsx
export default async function AuthLayout({ children }) {
  const user = await getUser(); // Cache HIT — access.ts already resolved this

  return (
    <div>
      <nav>{user.name}</nav>
      <main>{children}</main>
    </div>
  );
}
```

## Nested Access Checks

Deeper segments can add their own `access.ts` for additional authorization:

```ts
// app/(authenticated)/projects/[projectId]/access.ts
export default async function access(ctx: AccessContext) {
  const user = await requireUser(); // Cache HIT from parent
  const project = await getProject(ctx.params.projectId);
  if (!project) deny(404);
  if (project.orgId !== user.orgId) deny(); // Default 403
}
```

Access runs root-to-leaf. If the root gate fails, deeper gates never execute.

## `deny()`

`deny()` produces the correct HTTP status code and renders the matching status file:

```ts
deny();      // 403 Forbidden (default)
deny(404);   // 404 Not Found
deny(401);   // 401 Unauthorized
```

## Slot Degradation

When `access.ts` denies a parallel route (slot), only that slot degrades — the page continues rendering. The denied slot renders its `denied.tsx` fallback:

```
app/
  dashboard/
    layout.tsx           # Receives { children, admin }
    page.tsx
    @admin/
      access.ts          # Checks admin role
      page.tsx           # Admin panel
      denied.tsx         # Shown when access fails
```

If the user isn't an admin, `@admin` renders `denied.tsx` while the rest of the dashboard loads normally.

## `AccessContext`

```ts
interface AccessContext {
  params: Record<string, string>;
  searchParams: T; // Typed if search-params.ts exists
}
```

`cookies()` and `headers()` are imported from `@timber-js/app/server` — they work the same in `access.ts` as everywhere else.

## Middleware vs Access

| | `middleware.ts` | `access.ts` |
|--|----------------|-------------|
| Runs | Before rendering | Inside React tree |
| Purpose | Lightweight checks, redirects, headers | Auth, data gating |
| `React.cache` | Not available | Active — shares scope with layout |
| Slots | N/A | Graceful degradation |
